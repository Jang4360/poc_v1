# 06 검증, 시연, V2 경계

## 워크스트림

`POC 검증 흐름과 V2 경계 고정`

## 목표

현재 스프린트 결과로 어떤 방식의 MVP POC를 검증하고 시연할지 정하고, 자동화나 확장 기능은 V2 범위로 명확히 분리한다.

## 범위

- 최소 시연 흐름 정의
- smoke 및 수동 QA 기준 정리
- 현재 단계에서 구현하지 않을 항목 정의
- V2 backlog 경계 정리

## 비목표

- 운영 자동화 구현
- 완전한 Android 앱 완성
- 계정 로그인 구현

## 입력 근거

- `poc/`
- `.ai/EVALS/smoke-checklist.md`
- `docs/prd.md`
- `docs/erd.md`
- `etl/raw/`

## 성공 기준

- [ ] 현재 POC에서 시연 가능한 사용자 흐름이 정의되어 있다.
- [ ] smoke와 수동 검증 체크리스트가 실행 순서에 맞춰 정리되어 있다.
- [ ] 자동화 및 확장 기능이 V2로 명확히 분리되어 있다.

## 구현 계획

- [ ] 최소 시연 흐름을 다음 순서로 고정한다.
  - 주변 시설 조회
  - 일반 경로 탐색
  - 시각/이동성 기준 경로 차이 확인
  - 접근성 대중교통 보조 경로 확인
- [ ] smoke 순서를 `DB 기동 -> 정규 네트워크 확인 -> 보강 ETL 확인 -> GraphHopper 확인 -> API 확인`으로 정리한다.
- [ ] 시연용 샘플 좌표와 샘플 시설을 확정한다.
- [ ] V2로 넘길 항목을 명시한다.
  - ETL 자동 감지/재실행 자동화
  - 계정 및 즐겨찾기
  - LLM 대화형 UI
  - 익명 위험 제보
  - 완전한 Android 앱
  - 운영용 캐시/모니터링 고도화

## 검증 계획

- [ ] 단계별 실패 시 어떤 대체 검증을 할지 정리한다.
- [ ] `scripts/smoke.sh`가 현재 프로젝트 명령을 반영하는지 확인한다.
- [ ] 필수 데이터가 비어 있을 때 fallback 시연 시나리오를 준비한다.

## 위험 및 열린 질문

- 실제 UI가 없는 상태에서는 API와 데이터 기준 검증으로 시연이 축소될 수 있다.
- 데이터 공백이나 미매칭이 남아 있으면 시연 구간을 좁혀야 할 수 있다.
