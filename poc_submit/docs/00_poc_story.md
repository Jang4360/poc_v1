# 보행약자 지도 PoC 진행 과정과 시행착오

작성일: 2026-04-27

이 문서는 이번 PoC를 진행하면서 어떤 데이터를 보고, 어떤 시행착오를 거쳤고, 왜 지금의 기준으로 정리했는지 남기기 위한 기록이다. 단순히 최종 파일 목록을 설명하는 문서가 아니라, 팀원이 이후 작업을 이어받을 때 "왜 이 데이터는 남겼고, 왜 이 데이터는 뺐는지"를 이해할 수 있게 하는 것이 목적이다.

## 1. 처음 목표

처음 목표는 단순했다. 부산광역시 지도 위에 보행약자에게 의미 있는 정보를 표시하고 싶었다.

초기에는 다음 정도를 생각했다.

| 대상 | 처음 기대 |
|---|---|
| 계단 | 보행 중 피해야 할 계단 위치를 지도에 표시 |
| 횡단보도 | 보행 경로와 연결되는 횡단보도 표시 |
| 편의시설 | 보행약자에게 유용한 시설 표시 |
| 접근성 | 경사로, 엘리베이터, 장애인 화장실 같은 속성 표시 |
| 도로 | 경사도, 노면, 폭을 보행 난이도로 표현 |

하지만 진행하면서 알게 된 것은, 데이터를 많이 모으는 것보다 "무엇을 서비스에 남길지"가 훨씬 어려웠다는 점이다. 공공데이터라고 해서 그대로 서비스에 쓸 수 있는 것이 아니었고, 이름이 맞아 보여도 실제로는 내부시설, 사유지, 오래된 명칭, 위치 오류, 중복 데이터가 섞여 있었다.

그래서 PoC의 성격이 중간에 바뀌었다.

| 처음 생각 | 실제로 필요했던 일 |
|---|---|
| 데이터를 지도에 표시하면 된다 | 데이터가 서비스 목적에 맞는지 검증해야 한다 |
| 공식 데이터면 믿을 수 있다 | 공식 데이터도 목적지/접근성/내부시설을 구분해야 한다 |
| 계단은 계단이면 된다 | 실제 보행 경로에서 의미 있는 계단인지 봐야 한다 |
| 포장/비포장만 보면 된다 | 경사, 폭, 노면을 같이 봐야 한다 |
| 로드뷰 AI로 다 확인할 수 있다 | 로드뷰 결측, 방향, 가림, 검은 화면 때문에 보조 검증으로 써야 한다 |

## 2. 지도 PoC부터 시작

초기에는 HTML에 지도를 직접 띄우고, 부산광역시를 기본 위치로 보이게 했다. SDK로 지도를 띄울지, 단순 HTML/Leaflet으로 빠르게 갈지 고민했지만, PoC에서는 빠르게 움직이고 레이어를 직접 붙이는 것이 중요했기 때문에 HTML 기반으로 시작했다.

이 선택은 PoC에는 맞았다.

| 선택 | 이유 |
|---|---|
| 단일 `index.html` | 팀원이 바로 열어볼 수 있음 |
| Leaflet 기반 | GeoJSON/JS 데이터를 빠르게 얹기 쉬움 |
| JS 데이터 파일 분리 | 데이터가 많아져도 레이어별 관리 가능 |
| 구 단위 lazy-load | 부산 전체 도로 데이터를 한 번에 올리면 무거워짐 |

다만 시간이 지나면서 레이어가 많아졌다. 계단, 횡단보도, 편의시설, 도로 polygon, 경사/노면/폭, 검증 후보가 모두 들어오면서 UI가 복잡해졌고, 범례와 필터명을 정리해야 했다.

결론은 다음이다.

- 지도는 PoC 확인용으로는 충분하다.
- 레이어가 많아질수록 기본 ON/OFF와 범례가 중요하다.
- 최종 서비스에서는 모든 데이터를 동시에 보여주면 안 되고, 사용 목적에 맞게 단계적으로 보여줘야 한다.

## 3. 편의시설 데이터에서 겪은 문제

편의시설은 처음에 가장 쉬워 보였지만, 실제로는 가장 많은 기준 정리가 필요했다.

처음에는 장애인 편의시설 공공데이터와 기존 장소 데이터, 접근성 데이터를 합치면 된다고 생각했다. 하지만 원본을 합쳐보니 13,570건 규모가 되었고, 이 안에는 서비스 목적지로 쓰기 어려운 데이터가 많이 섞여 있었다.

대표적인 문제는 다음과 같았다.

| 문제 | 실제 사례 또는 현상 |
|---|---|
| 카테고리가 너무 넓음 | 기존 `BARRIER_FREE_FACILITY` 하나로는 의료, 복지, 공공기관을 구분하기 어려움 |
| 이름이 모호함 | 건물명, 시장명, 지하상가명, 과거 상호가 섞임 |
| 화장실 범위가 애매함 | 공중화장실, 시설 내부 화장실, 장애인 전용 화장실 정보가 섞임 |
| 전동보장구 충전소 위치가 애매함 | 장소로 볼지, 접근성 속성으로 볼지 결정 필요 |
| POI 최신성 문제 | POI에는 과거명이 남아 있고, 카카오에는 최신 상호가 잡히는 경우가 있음 |
| 카카오 누락 문제 | 공중화장실/충전소처럼 공공성 강한 데이터는 카카오에서 안 잡히는 경우가 있음 |

여기서 가장 중요한 전환점은 "공식 데이터니까 다 유지"가 아니라 "보행약자가 실제로 찾아갈 수 있는 목적지인가"를 기준으로 보게 된 것이다.

## 4. 접근성이라는 말이 애매했다

초기에 `accessible`이라는 표현이 계속 애매했다. "접근 가능한 실/객실", "접근 가능한 출입구", "장애인 이용 가능 객실" 같은 표현은 일관성이 떨어졌다.

그래서 UI에서는 추상적인 accessible보다 사용자가 이해할 수 있는 기능 단위로 표현하기로 했다.

| 내부 featureType | UI 표현 |
|---|---|
| `ramp` | 경사로 |
| `elevator` | 엘리베이터 |
| `stepFree` | 단차 없음 |
| `accessibleToilet` | 전용 화장실 |

서비스 문맥은 "장애인 전용"보다 더 넓게 "보행약자" 관점으로 잡았다. 휠체어 이용자뿐 아니라 유모차, 고령자, 보행보조기 이용자에게도 의미가 있기 때문이다.

## 5. 장소와 접근성 분리

가장 헷갈렸던 부분은 화장실과 전동보장구 충전소였다.

처음에는 둘 다 "접근성"으로 볼 수 있지 않나 싶었다. 하지만 데이터를 보면서 결론을 바꿨다.

| 대상 | 최종 판단 | 이유 |
|---|---|---|
| 공중/개방 화장실 | 장소 `TOILET` | 사용자가 목적지로 찾아갈 수 있음 |
| 장애인 화장실 유무 | 접근성 `accessibleToilet` | 어떤 장소의 속성으로 붙는 정보 |
| 전동보장구 충전소 | 장소 `CHARGING_STATION` | 충전 자체가 방문 목적이 될 수 있음 |
| 충전기 있음 feature | 현재 미생성 | 특정 host place의 부가 속성으로 볼 근거가 부족함 |

이 기준을 잡으면서 ERD도 명확해졌다.

| 테이블 | 역할 |
|---|---|
| `places` | 장소 마스터 |
| `place_accessibility_features` | 장소별 접근성 속성 |

최종적으로 `places 1 : N place_accessibility_features` 관계를 유지했고, `UNIQUE(placeId, featureType)` 제약에 맞게 데이터를 정리했다.

## 6. 최종 편의시설 카테고리까지 간 과정

처음에는 공공데이터의 70여 개 카테고리를 그대로 보려고 했다. 하지만 서비스에서 그대로 보여주기에는 너무 세분화되어 있었고, 반대로 기존 ERD의 `BARRIER_FREE_FACILITY`는 너무 넓었다.

그래서 실제 서비스 목적지로 의미 있는 카테고리만 남기고 합쳤다.

최종 카테고리는 다음이다.

| category | 의미 |
|---|---|
| `TOILET` | 공중/시설 내 화장실 |
| `RESTAURANT` | 음식·카페 |
| `TOURIST_SPOT` | 관광지 |
| `ACCOMMODATION` | 숙박 |
| `CHARGING_STATION` | 전동보장구 충전소 |
| `HEALTHCARE` | 병원/의원/보건소 |
| `WELFARE` | 복지시설/경로당/요양시설 |
| `PUBLIC_OFFICE` | 주민센터/공공기관/파출소/우체국/공단 |

기존에 있던 `BARRIER_FREE_FACILITY`는 최종 서비스 카테고리로는 제거했다. 다만 `sourceDataset=barrier_free_facility` 같은 원천 식별자는 남겼다. 이것은 카테고리가 아니라 "어느 원본에서 왔는지"를 추적하기 위한 값이다.

## 7. 편의시설 검증 방식

편의시설 검증은 한 번에 끝나지 않았다. 여러 번 기준을 바꿔가며 좁혀갔다.

최종 흐름은 다음과 같다.

| 단계 | 한 일 |
|---|---|
| 1 | 원본 장소와 접근성 원본을 전체 병합 |
| 2 | 공공데이터 실제 시설 유형을 참고해 카테고리 후보 정리 |
| 3 | 보행약자 서비스에 필요한 카테고리만 채택 |
| 4 | ERD 기준으로 `places`, `place_accessibility_features` 분리 |
| 5 | 카카오맵 우선, POI 보조 방식으로 이름/존재 여부 확인 |
| 6 | 이름이 과거명인 경우 카카오 또는 POI 기준으로 보정 |
| 7 | 둘 다 근거가 없는 장소는 제거 후보로 분류 |
| 8 | 화장실/음식점/복지/의료/공공기관 등 카테고리별 수동 규칙 적용 |
| 9 | 회사/비방문 목적지 5건, 우선 수동검증 위험 후보 25건 제거 |
| 10 | `RENAME_REVIEW` 978건을 카카오맵 기준 표시명으로 보정 |
| 11 | 최종 3,432개 장소와 5,337개 접근성 row 생성 |

카카오와 POI는 서로 역할이 달랐다.

| 데이터 | 장점 | 한계 |
|---|---|---|
| 카카오 | 최신 상호/현재 장소 확인에 유리 | 공중화장실/충전소 같은 공공 데이터가 누락될 수 있음 |
| POI | 공공성 시설이나 오래된 장소 확인에 도움 | 과거명, 폐업, 위치 부정확 가능성 |
| 공공데이터 | 접근성 속성의 근거로 중요 | 서비스 목적지로 바로 쓰기에는 내부시설/범위 문제 있음 |

그래서 최종 원칙은 "일반 장소는 카카오 우선, POI는 보조, 접근성 속성은 공공데이터 근거 유지"로 정리했다.

## 8. 화장실에서 가장 많이 흔들렸다

화장실은 계속 기준이 흔들렸다. 이유는 "화장실"이라는 이름만 보고는 사용자가 실제로 찾아갈 수 있는지 알기 어렵기 때문이다.

예를 들어 다음 같은 것들이 섞였다.

| 유형 | 문제 |
|---|---|
| 지하철역/철도역/터미널 화장실 | 거의 모든 역에 존재해 별도 목적지로 표시하면 노이즈가 큼 |
| 음식점/카페 내부 화장실 | 고객 전용일 가능성이 높음 |
| 주유소/호텔/빌딩/아파트 화장실 | 보행약자 목적지로 안내하기 애매함 |
| 학교/군부대/예비군 관련 화장실 | 일반 접근성이 낮음 |
| 시장/공원/관광지 화장실 | 실제로 공중 이용 가능성이 있어 유지 가능 |

최종 기준은 다음이다.

| 기준 | 처리 |
|---|---|
| 독립 공중화장실 | 유지 |
| 공원/관광지/시장 등 공공 이용 가능성이 있는 화장실 | 유지 |
| 시설 내부 장애인 화장실 유무 | `accessibleToilet`로 표현 |
| 지하철역/철도역/터미널 내부 화장실 | 제거 |
| 음식점/카페/주유소/호텔/빌딩/아파트 내부 화장실 | 제거 |
| 학교/군부대/예비군/사유시설 화장실 | 제거 |

결론은 "화장실이라는 단어가 있는가"가 아니라 "보행약자가 독립 목적지로 찾아가도 되는가"다.

## 9. 계단 데이터에서 겪은 문제

계단은 수치지형도 데이터에서 시작했다. 원천 후보는 3,744건이었다.

처음에는 계단 전체를 지도에 표시하면 될 것 같았다. 하지만 지도에서 확인해보니 다음 문제가 있었다.

| 문제 | 설명 |
|---|---|
| 내부 계단 | 학교, 아파트, 건물 내부 계단 가능성 |
| 사유지 계단 | 일반 보행 경로가 아닐 수 있음 |
| 도로와 연결되지 않는 계단 | 경로 탐색에 반영하기 어려움 |
| 공원/산책로 계단 | 보행 경로일 수 있지만 도로 기반 판단만으로는 애매함 |
| 위치 오차 | 수치지형도 좌표와 실제 로드뷰 위치가 다를 수 있음 |

그래서 계단을 바로 확정하지 않고 우선순위를 만들었다.

| 우선순위 | 의미 |
|---|---|
| P1 | 도로/횡단보도/공공 맥락과 가까워 우선 검토 |
| P2 | 도로와 가깝지만 추가 맥락 확인 필요 |
| P3 | 가능성은 있으나 주변 맥락이 약함 |
| P4 | 내부/사유지/저연결성 가능성이 높음 |

초기 우선순위는 P1 728건, P2 138건, P3 862건, P4 2,016건이었다.

## 10. 계단을 전수 검토하려다 방향을 바꾼 이유

처음에는 로드뷰로 전부 검증하려고 했다. 로드뷰를 8방향으로 돌리고, 이미지를 AI에 넣어 계단 여부를 판단하는 방식까지 테스트했다.

하지만 실제로 해보니 한계가 명확했다. 특히 AI 결과에서 `NOT_STAIR`가 대부분으로 나왔다. 이것을 그대로 "계단 없음"으로 받아들이기에는 위험했다. 계단이 실제로 없어서 안 잡힌 것인지, 로드뷰 방향이 맞지 않아서 안 보인 것인지, 로드뷰 촬영 위치가 계단과 어긋난 것인지 구분하기 어려웠기 때문이다.

| 문제 | 설명 |
|---|---|
| 로드뷰 결측 | 골목, 공원, 사유지 주변은 로드뷰가 없을 수 있음 |
| 방향 문제 | 계단이 있어도 카메라 방향이 맞지 않으면 안 보임 |
| 검은 화면 | 로딩이 끝나기 전에 캡처하거나 로드뷰 자체가 실패하는 경우 |
| 가림 문제 | 차량, 벽, 나무, 건물에 계단이 가려짐 |
| 비용/시간 | 전체 계단을 이미지로 검증하면 시간이 너무 오래 걸림 |

로드뷰 AI 결과는 다음처럼 나왔다.

| 결과 | 건수 |
|---|---:|
| `CONFIRMED_STAIR` | 202 |
| `NOT_STAIR` | 660 |
| `NO_ROADVIEW` | 92 |
| `PRIVATE_OR_INTERNAL` | 18 |
| `UNCLEAR` | 6 |

이 결과로 알게 된 것은, 로드뷰 AI는 보조 검증에는 좋지만 최종 정답으로 쓰기 어렵다는 점이다. 특히 `NOT_STAIR`가 많다는 사실은 "계단이 없다"가 아니라 "로드뷰에서 확인하지 못했다"에 가깝다.

그래서 계단 로드뷰 자동 검증 방식은 사실상 폐기하기로 했다. 로드뷰 캡처와 AI 판정 결과는 시행착오 기록과 참고 근거로만 남긴다. 최종 채택/제외 기준은 AI가 보인다고 한 것만 쓰는 방식이 아니라, 우선 도로/횡단보도/편의시설과의 거리, 주변 공공 보행 맥락, 학교/사유지/아파트 단지 내부 가능성 같은 제외 조건을 기준으로 선별하는 방식으로 바꿨다.

## 11. 계단 최종 방향

계단은 다음 방식이 현실적이라고 판단했다.

| 단계 | 처리 |
|---|---|
| 1 | 수치지형도 계단 전체를 후보로 둠 |
| 2 | 도로, 횡단보도, 공공시설, 주거/사유지 맥락으로 1차 분류 |
| 3 | `LIKELY_ADOPT`, `LIKELY_EXCLUDE`, `MANUAL_REVIEW`로 나눔 |
| 4 | `LIKELY_ADOPT`는 지도 반영 후보로 둠 |
| 5 | `MANUAL_REVIEW`는 사람이 지도에서 직접 확인 |
| 6 | 로드뷰 AI 자동 검증은 최종 기준에서 제외하고 시행착오로만 보존 |

여기서 핵심은 로드뷰 AI 결과보다 거리/맥락 기반 선별을 우선했다는 점이다. 도로와 가깝고 횡단보도나 공공시설 맥락이 있으면 채택 가능성을 높게 보고, 학교·군부대·사유시설·단지 내부처럼 일반 보행 경로로 보기 어려운 맥락은 제외 또는 수동검토로 내렸다.

다음 검증은 자동화가 아니라 수작업으로 진행한다. 우선 강서구부터 편의시설과 계단을 하나씩 확인하고, 채택/제외/수정 사유를 기록한 뒤 다른 구로 확장한다.

2차 판정 결과는 다음이다.

| 판정 | 건수 | 의미 |
|---|---:|---|
| `LIKELY_ADOPT` | 1,214 | 공개 보행로 계단일 가능성이 높음 |
| `LIKELY_EXCLUDE` | 661 | 내부/사유지/저연결성 가능성이 높음 |
| `MANUAL_REVIEW` | 742 | 데이터만으로 확정하기 어려움 |

P4도 일부는 채택 가능성으로 승격되었다. 다만 P4는 원래 저연결성/내부 가능성이 높기 때문에, 최종 서비스에 넣기 전에는 더 조심해서 봐야 한다.

## 12. 횡단보도 데이터 시행착오

횡단보도는 기존 데이터에 결측이 있다는 것을 발견했고, 한때 TMap에서 횡단보도 데이터를 가져와 지도에 표시했다.

하지만 이후 적용하지 않기로 결정했다. 이유는 다음과 같다.

| 이유 | 설명 |
|---|---|
| 기준 혼선 | 기존 수치지형도/공공데이터와 TMap 데이터의 기준이 달랐음 |
| 검증 부담 | 어떤 것이 더 정확한지 판단하려면 별도 비교 검증이 필요 |
| PoC 범위 초과 | 당시 우선순위는 편의시설, 계단, 도로 경사/노면/폭이었다 |

결론은 "일단 기존 채택본 기준을 유지하고, TMap은 추후 별도 검증 후 반영"이다.

## 13. 도로 polygon 시행착오

팀원 작업 방향과 맞춰 도로 중심선을 polygon처럼 표시하는 작업도 진행했다. 처음에는 모든 도로를 polygon으로 표시해서 도로 폭을 시각적으로 보여주려고 했다.

하지만 지도에 올려보니 생각보다 보기 좋지 않았다. 도로 polygon이 겹치고, 교차점에서 이상한 면이 생기고, 사용자가 기대하는 "도로 면"과 달라 보였다.

이 과정에서 얻은 결론은 다음이다.

| 시행착오 | 결론 |
|---|---|
| 도로 중심선+폭으로 polygon 생성 | 빠른 PoC는 가능하지만 시각적으로 어색할 수 있음 |
| 전체 부산 도로를 한 번에 표시 | 너무 무겁고 UX가 나빠짐 |
| 해운대구만 먼저 표시 | 검증에는 좋지만 최종적으로는 모든 구를 구 단위로 나눠야 함 |
| 도로 polygon을 진짜 도로 면처럼 기대 | 중심선 buffer는 실제 도로 경계 polygon과 다름 |

최종적으로는 도로 데이터를 구 단위로 분리하고, 선택한 구만 로드하는 방식으로 정리했다.

## 14. 경사/노면/폭 기준

경사도, 노면, 폭은 서비스적으로 중요하지만, 단일 기준으로 자르면 위험하다.

처음에는 포장/비포장만 보면 될지 고민했다. 하지만 비포장이라고 해서 무조건 통행 불가가 아니고, 포장이라고 해서 보행약자에게 항상 안전한 것도 아니다.

그래서 최종적으로는 세 요소를 함께 보는 방향으로 잡았다.

| 요소 | 의미 |
|---|---|
| 경사도 | 높을수록 보행 부담 증가 |
| 폭 | 좁을수록 휠체어/유모차/보행보조기 이동 부담 증가 |
| 노면 | 비포장/미분류는 부담 요소지만 단독 차단 기준은 아님 |

서비스 등급은 다음처럼 정리했다.

| 등급 | UI 표현 | 의미 |
|---|---|---|
| `LOW` | 무난 | 보행 부담이 낮은 구간 |
| `MEDIUM` | 주의 | 일부 사용자에게 부담 가능 |
| `HIGH` | 어려움 | 보행약자에게 부담이 큰 구간 |
| `VERY_HIGH` | 회피 권장 | 대체 경로가 있으면 우회 권장 |

중요한 점은 `VERY_HIGH`도 절대 통행 불가가 아니라는 것이다. 서비스에서는 경로 탐색 비용을 높이거나 지도에서 강하게 표시하는 방식이 더 맞다.

## 15. 최종 제출본을 따로 만든 이유

원본 `poc` 폴더는 약 96GB였다. 대부분은 로드뷰 자동화 과정에서 생긴 Chrome 프로필과 캡처 이미지였다.

처음에는 원본 폴더에서 불필요한 것을 삭제하려고 했지만, 원본을 직접 삭제하면 위험했다. 그래서 제출용 폴더를 따로 만들었다.

| 원칙 | 이유 |
|---|---|
| 원본 `poc`는 보존 | 나중에 원본/중간 산출물을 다시 확인할 수 있음 |
| `poc_submit`만 제출 | 필요한 최종 산출물만 포함 |
| 대용량 캡처 제외 | 지도 실행과 팀 공유에는 필요 없음 |
| 로드뷰 결과 CSV만 유지 | 검증 근거는 남기되 이미지 원본은 제외 |

제출본에는 다음을 포함했다.

| 포함 | 이유 |
|---|---|
| `index.html` | 지도 실행 |

제출본에서 제외한 대표적인 항목은 다음이다.

| 제외 | 이유 |
|---|---|
| `chrome_profiles/` | 자동화 브라우저 프로필, 약 86GB |
| `captures/` | 로드뷰 원본 이미지, 약 7.7GB |

## 19. 이번 PoC의 교훈

데이터 PoC는 "지도에 많이 찍기"가 아니라 "서비스에 남겨도 되는 데이터를 고르는 과정"이었다.

이번에 특히 크게 배운 점은 다음이다.

- 공식 데이터도 그대로 쓰면 안 된다.
- 카카오와 POI는 서로 보완 관계다.
- 접근성은 단어가 아니라 기능으로 표현해야 한다.
- 장소와 접근성 속성은 분리해야 한다.
- 화장실은 독립 목적지인지 내부 속성인지 반드시 구분해야 한다.
- 계단은 실제 보행 경로와 연결되는지 봐야 한다.
- 로드뷰 AI는 강력하지만, 최종 판정 자동화로 쓰기에는 결측과 방향 문제가 크다.
- 경사/노면/폭은 절대 차단 기준이 아니라 보행 부담 기준으로 써야 한다.
- 문서화 없이 계속 진행하면 나중에 왜 그렇게 했는지 설명할 수 없다.

이 문서는 그 시행착오를 다시 반복하지 않기 위한 기록이다.
