# 06. 제출 패키지 구성 정책

이 문서는 원본 PoC 작업 폴더에서 제출본 `poc_submit`을 따로 분리한 기준을 정리한다.

## 배경

원본 `poc` 폴더는 약 96GB였다. 크기가 커진 주된 이유는 로드뷰 자동화 검증 과정에서 생성된 Chrome 프로필, 로드뷰 캡처 이미지, AI 입력용 변환 이미지 때문이었다.

이 파일들은 시행착오와 검증 과정에서는 필요했지만, 팀원이 지도와 최종 산출물을 확인하는 제출본에는 필요하지 않았다.

따라서 원본은 보존하고, 제출에는 필요한 파일만 담은 `poc_submit` 폴더를 별도로 구성했다.

## 제출본 구성 원칙

| 원칙 | 설명 |
|---|---|
| 지도 실행 가능 | `index.html`을 열면 현재 PoC 지도를 확인할 수 있어야 한다 |
| 최종 CSV 포함 | DB 반영 후보인 `places_erd.csv`, `place_accessibility_features_erd.csv`를 포함한다 |
| 기준 문서 포함 | 왜 남기고 왜 제거했는지 설명하는 문서를 포함한다 |
| 검증 근거 최소 포함 | 로드뷰 검증은 결과 CSV만 포함하고 원본 이미지는 제외한다 |
| 대용량 중간 산출물 제외 | 재생성 가능하거나 제출 검토에 직접 필요 없는 파일은 제외한다 |
| 원본 보존 | 원본 작업 폴더는 삭제하지 않고 로컬에 남긴다 |

## 포함 대상

| 대상 | 이유 |
|---|---|
| `index.html` | 지도 실행 본체 |
| `assets/` | 지도 표시용 데이터 |
| `data/adopted/` | 최종 채택 데이터 |
| `data/final/` | 최종 스냅샷과 요약 |
| `data/reports/stair_roadview_validation/ai_validation_results.csv` | 계단 로드뷰 AI 결과 |
| `data/reports/stair_roadview_validation/capture_manifest.csv` | 계단 로드뷰 캡처 상태 요약 |
| `docs/` | 정책, 로그, 시행착오, 제출 기준 문서 |
| `scripts/` | 재현 또는 검증용 스크립트 |

## 제외 대상

| 대상 | 제외 이유 |
|---|---|
| `chrome_profiles/` | 로드뷰 자동화 브라우저 프로필. 지도 실행/최종 검토에 불필요하며 매우 큼 |
| `captures/` | 로드뷰 원본 캡처 이미지. 결과 CSV가 있으므로 제출본에서는 제외 |
| `jpeg_inputs/` | AI 검증용 변환 이미지. 재검증이 필요할 때만 원본 폴더에서 확인 |
| `roadview_validation_test/` | 로드뷰 테스트 과정 부산물 |
| `data/source/` | 원본 데이터. 제출본은 최종 결과 공유 목적이므로 제외 |
| `data/staging/` | 중간 산출물. 재생성 가능하고 용량이 큼 |
| `data/geo/` | 원본 GIS 데이터. 제출본 지도 실행에는 불필요 |
| `archive/` | 중단된 실험/이전 산출물 |
| `data/reports/facility_validation/` | 편의시설 중간 검증 리포트. 최종 반영 결과는 `data/adopted`, `data/final/facilities`, 문서에 요약 |

## 로드뷰 파일을 제외한 이유

계단 검증 과정에서 로드뷰를 8방향으로 캡처하고 AI로 계단 여부를 판정했다. 하지만 결과적으로 `NOT_STAIR`가 대부분이었고, 이것을 최종 제외 근거로 그대로 쓰기 어렵다는 결론을 냈다.

주요 이유는 다음과 같다.

| 이유 | 설명 |
|---|---|
| 위치 오차 | 계단 좌표와 로드뷰 촬영 위치가 정확히 일치하지 않음 |
| 방향 문제 | 계단이 있어도 카메라 방향 또는 촬영 지점에 따라 보이지 않을 수 있음 |
| 로드뷰 결측 | 골목, 공원, 단지 내부, 사유지 주변은 로드뷰가 없을 수 있음 |
| 검은 화면/로딩 실패 | 일부 캡처가 정상적으로 생성되지 않음 |
| 가림 문제 | 차량, 벽, 나무, 건물 때문에 계단이 가려질 수 있음 |

따라서 로드뷰 원본 이미지를 제출본에 모두 포함하는 것은 비용 대비 효용이 낮다. 더 중요하게는, 계단 로드뷰 자동 검증 자체를 최종 기준으로 쓰지 않기로 했다. 제출본에는 결과 CSV만 시행착오 기록으로 포함하고, 실제 채택 기준은 거리/맥락 기반 선별과 제외 규칙으로 문서화했다.

## 계단 선별 기준과 제출본의 관계

계단은 로드뷰 AI가 "계단 있음"이라고 한 것만 채택한 데이터가 아니다.

현재 기준은 다음에 가깝다.

```text
도로/횡단보도/편의시설과 가까운 계단 = 채택 가능성 증가
학교/군부대/사유시설/아파트 단지 내부 맥락 = 제외 또는 수동검토
로드뷰 AI = 최종 기준에서 제외, 시행착오 기록으로만 보존
```

즉 제출본의 계단 산출물은 "AI 전수 판정 결과"가 아니라 "거리와 주변 맥락으로 우선 선별한 후보 데이터"로 봐야 한다.

## 다음 검증 방식

다음 단계는 자동화가 아니라 수작업 검증이다. 강서구부터 시작해서 편의시설과 계단을 함께 확인한다.

| 순서 | 작업 |
|---:|---|
| 1 | 강서구 편의시설을 지도에서 하나씩 확인 |
| 2 | 같은 구의 계단 후보를 도로/횡단보도/편의시설과 함께 확인 |
| 3 | 학교/군부대/사유시설/아파트 내부 가능성이 큰 계단은 제외 또는 보류 |
| 4 | 채택/제외/수정 사유를 기록 |
| 5 | 검증 방식이 안정되면 다른 구로 확장 |

## 좌표 수정 반영 방식

수동 검증 중 좌표가 틀린 편의시설을 발견하면 지도 팝업에서 카카오 지도/로드뷰 URL을 입력해 `FIX` 상태로 저장한다. 이 값은 브라우저 `localStorage`에 우선 저장되며, `현재 구 검증 CSV 내보내기`로 CSV 파일로 분리한다.

실제 최종 산출물에는 `scripts/apply_manual_review_coordinate_fixes.py`로 반영한다.

```powershell
python scripts/apply_manual_review_coordinate_fixes.py --review-csv "C:\path\manual_review_강서구_YYYYMMDDHHMMSS.csv" --dry-run
python scripts/apply_manual_review_coordinate_fixes.py --review-csv "C:\path\manual_review_강서구_YYYYMMDDHHMMSS.csv"
```

`--dry-run`은 변경 예정 보고서만 만들고 원본 CSV/JS는 수정하지 않는다. 실제 반영 후에는 `data/reports/manual_review/`에 적용 보고서가 남는다.

## 제출본과 원본의 역할

| 구분 | 역할 |
|---|---|
| 원본 `poc` | 원본 데이터, 중간 산출물, 대용량 검증 파일 보관 |
| 제출본 `poc_submit` | 팀 공유, 지도 실행, 최종 결과 검토 |

제출본만으로 지도 확인과 최종 CSV 검토는 가능하다. 다만 원본 데이터부터 전체 과정을 다시 재현하려면 원본 `poc` 폴더의 제외 파일들이 필요하다.

## 현재 제출본 크기

현재 제출본은 약 420MB다. 대부분은 전체 부산 도로 polygon과 경사/노면/폭 지도 데이터가 차지한다.

추가로 더 줄여야 한다면 다음 순서로 검토할 수 있다.

| 순서 | 방법 | 영향 |
|---:|---|---|
| 1 | 도로 polygon 레이어 제외 | 지도에서 도로 면 표시 불가 |
| 2 | 경사/노면/폭 전체 구 데이터 일부 제외 | 선택 구 외 데이터 확인 불가 |
| 3 | scripts 제외 | 재현성 낮아짐 |
| 4 | docs의 참고 PDF 제외 | 문서 근거 일부 약해짐 |

현재는 팀 공유와 지도 확인을 위해 420MB 수준을 유지하는 것이 적절하다.
