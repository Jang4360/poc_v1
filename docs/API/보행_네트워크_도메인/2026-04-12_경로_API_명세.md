> **작성일:** 2026-04-12  
> **작성자:** 김응서  
> **최종 수정일:** 2026-04-22

---

## 1. API 목록

| 우선순위 | API | 설명 |
| --- | --- | --- |
| P1 | `POST /routes/search` | 장애 유형과 경로 옵션 기준 경로 탐색 |
| P2 | `POST /routes/reroute` | 경로 이탈 시 현재 위치 기준 재탐색 |
| P3 | `POST /route-logs` | 내비게이션 종료 후 선택적 이동 로그 업로드 |

---

## 2. 설계 원칙

- `routeOptions` 배열 대신 `routeOption` 단일 값을 사용한다.
- `routeOption` 후보값은 `SAFE`, `SHORTEST`, `PUBLIC_TRANSPORT`만 사용한다.
- 내부 GraphHopper 프로필 이름(`visual_safe`, `visual_fast`, `wheelchair_safe`, `wheelchair_fast`)은 외부 API enum에 직접 노출하지 않는다.
- 응답의 보행 세그먼트 상태값은 boolean 대신 ERD 기준 상태 필드로 응답한다.
- `POST /routes/reroute`는 실시간 GPS 추적 결과를 받아 재탐색하는 API이며, `POST /route-logs`에 의존하지 않는다.
- `POST /route-logs`는 사후 분석용 저장 API다. 실시간 재탐색 트리거 용도로 사용하지 않는다.

## 3. 경로 탐색

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/routes/search` |
| 인증 | 불필요 |
| 설명 | 출발지, 도착지, 장애 유형, 경로 옵션을 기준으로 접근 가능한 경로 옵션을 조회한다. 도보 경로는 GraphHopper 결과를, 대중교통 경로는 ODsay/BIMS/지하철 접근성 데이터를 조합한 결과를 사용한다. |

### 비고

- `routeOption`은 단일 값으로 받되, 값이 없으면 서버가 `SAFE`, `SHORTEST`, `PUBLIC_TRANSPORT` 전체를 병렬 계산해 반환할 수 있다.
- 프론트에서 탭 3개를 한 번에 띄워야 하면 `routeOption`을 생략한 한 번의 호출을 기본으로 사용한다.
- `segments`는 순서가 있는 리스트이며, 프론트는 각 세그먼트의 `geometry`를 이어서 지도 라인을 그린다.
- `PUBLIC_TRANSPORT`는 ODsay/BIMS/지하철 접근성 검증 결과에 따라 `UNAVAILABLE`로 반환될 수 있다.
- boolean 기반 필드(`hasStairs`, `hasCurbGap`, `hasSignal`, `hasAudioSignal`, `hasBrailleBlock`)는 더 이상 사용하지 않는다.
- 대중교통 세그먼트 필드는 해당 구간에서만 값이 채워지며, 도보 세그먼트에서는 `null`이다.

| HTTP Status | code | message | 발생 조건 |

## 4. 경로 이탈 재탐색

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P2 |
| Method | `POST` |
| URL | `/routes/reroute` |
| 인증 | 불필요 |
| 설명 | 클라이언트가 현재 위치를 기준으로 기존 내비게이션 경로에서 이탈했다고 판단했을 때, 현재 위치부터 목적지까지 동일 옵션으로 다시 경로를 계산한다. |

### 비고

- 재탐색은 클라이언트 실시간 위치 추적 결과로 호출한다.
- 경로 이탈 판단은 클라이언트에서 수행한다. 예: 현재 위치가 활성 경로 geometry에서 20m 이상 이탈한 상태가 2회 연속 발생한 경우.
- `POST /route-logs`는 이 API의 선행 조건이 아니다.
- `PUBLIC_TRANSPORT` 재탐색도 동일한 API로 처리하되, 내부적으로는 ODsay/BIMS/지하철 접근성 검증을 다시 수행한다.

## 5. 실제 이동 경로 로그 수집

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P3 |
| Method | `POST` |
| URL | `/route-logs` |
| 인증 | 불필요 |
| 설명 | 내비게이션 종료 후 사용자가 동의한 경우에만 실제 이동 로그를 일괄 업로드한다. 실시간 재탐색 용도가 아니라 사후 품질 분석 및 데이터 검증 용도다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Content-Type | Y | `application/json` |

#### Query Parameter

> 없음

#### Body

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| disabilityType | string | Y | 이동 시점의 장애 유형. `VISUAL`, `MOBILITY` |
| routeOption | string | Y | 실제 사용한 옵션. `SAFE`, `SHORTEST`, `PUBLIC_TRANSPORT` |
| startedAt | string | Y | 내비게이션 시작 시각. ISO-8601 |
| endedAt | string | Y | 내비게이션 종료 시각. ISO-8601 |
| distanceMeter | number | N | 실제 이동 거리 |
| points | array | N | GPS 좌표 목록. 프론트에서 수집 가능할 때만 전송 |
| points[].sequence | int | Y | 좌표 순서 |
| points[].lat | number | Y | 위도 |
| points[].lng | number | Y | 경도 |
| points[].recordedAt | string | Y | GPS 기록 시각. ISO-8601 |
| points[].accuracyMeter | number | N | GPS 정확도. 단위 meter |

#### Request Example

```json
{
  "disabilityType": "MOBILITY",
  "routeOption": "SAFE",
  "startedAt": "2026-04-13T09:00:00+09:00",
  "endedAt": "2026-04-13T09:18:30+09:00",
  "distanceMeter": 950.4,
  "points": [
    {
      "sequence": 1,
      "lat": 35.1796,
      "lng": 129.0756,
      "recordedAt": "2026-04-13T09:00:02+09:00",
      "accuracyMeter": 8.5
    },
    {
      "sequence": 2,
      "lat": 35.1799,
      "lng": 129.0760,
      "recordedAt": "2026-04-13T09:00:07+09:00",
      "accuracyMeter": 7.9
    }
  ]
}
```

### Response

#### Success

- HTTP Status: `201 Created`

#### Data Fields

| 필드명 | 타입 | Null 가능 | 설명 |
| --- | --- | --- | --- |
| routeLogId | long | N | 저장된 로그 ID |

#### Success Example

```json
{
  "success": true,
  "data": {
    "routeLogId": 1
  },
  "message": null
}
```

### 비고

- 프론트가 GPS 포인트를 안정적으로 버퍼링할 수 없으면 `points` 없이 summary만 저장하도록 단순화할 수 있다.
- GPS 포인트는 매 순간 저장하지 않고, 기본적으로 `5초 이상 경과` 또는 `10m 이상 이동` 시에만 샘플링하는 것을 권장한다.
- `0.0003`도 차이는 부산 기준 대략 30m 전후이므로, 예시처럼 포인트를 너무 촘촘하게 찍을 필요는 없다.
- 장거리 이동에서는 업로드 전 클라이언트에서 단순 down-sampling을 적용하는 것을 권장한다.
- 이 API는 경로 품질 분석, 실패 케이스 회수, 향후 추천 데이터 보강용이다.
- 실시간 경로 이탈 감지는 이 API가 아니라 클라이언트의 위치 추적과 `/routes/reroute` 호출로 처리한다.
- 사용자 또는 기기 식별자는 저장하지 않는다.

### Error

| HTTP Status | code | message | 발생 조건 |
| --- | --- | --- | --- |
| 400 | INVALID_INPUT | 잘못된 입력입니다. | 필수값 누락, 좌표값 오류, `routeOption`/`disabilityType` 값 오류 |
| 500 | INTERNAL_ERROR | 서버 내부 오류가 발생했습니다. | 경로 로그 저장 중 서버 예외 |

## 6. 구현 흐름

1. 프론트는 `POST /routes/search`로 초기 경로를 조회한다.
2. 사용자가 선택한 경로 option과 geometry를 프론트가 메모리에 유지한다.
3. 프론트는 기기 GPS를 주기적으로 읽고, 활성 경로와의 이탈 여부를 로컬에서 판단한다.
4. 이탈 시 `POST /routes/reroute`로 현재 위치부터 목적지까지 동일 옵션을 다시 요청한다.
5. 내비게이션 종료 후 사용자가 동의한 경우에만 `POST /route-logs`로 사후 로그를 업로드한다.
